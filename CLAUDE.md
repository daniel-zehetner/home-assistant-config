# Home Assistant Configuration

This repo manages a Home Assistant instance configuration stored as version-controlled YAML files.

## Instance details

- Unit system: metric
- Currency: GBP
- Country: GB
- Timezone: Europe/London

## Directory structure

```
configuration.yaml          # Main HA entry point — loads subdirectories
secrets.yaml                # Sensitive values (gitignored — never commit)
.env                        # HA URL + API key (gitignored)
automations/                # !include_dir_merge_list  → list of automation objects
scripts/                    # !include_dir_merge_named → dict of script objects
scenes/                     # !include_dir_merge_list  → list of scene objects
dashboards/                 # Lovelace dashboard exports (YAML, one file per dashboard)
knx.yaml                    # KNX integration top-level (knx: !include knx.yaml)
knx/
  covers.yaml               # 15 Rollos — move_long/short, position, state
  lights.yaml               # 7 lights — address, state_address
  switches.yaml             # Gate + 9 irrigation valves
  binary_sensors.yaml       # Gate status + 8 irrigation valve statuses
  sensors.yaml              # 3 temperature sensors (Wohnzimmer, Schlafzimmer, Abstellraum)
  climate.yaml              # 3 climate entities (temperature_address only — add setpoints)
entities.yaml               # Live entity catalog (gitignored — regenerated by build_config.py)
build_config.py             # Re-fetches entities from HA and regenerates automation stubs
build_knx_yaml.py           # Rebuilds knx/ from ETS project via WebSocket API
export_dashboards.py        # Exports Lovelace dashboards via HA WebSocket API
fetch_knx.py                # Fetches raw KNX data from HA (ETS project, entity registry)
```

Add new topic files to the relevant directory; HA picks them up automatically on restart.

## Authoring rules

### IDs and aliases
- Every automation and scene **must** have a unique `id` (used by HA to track state across reloads)
- Use a descriptive `alias` — shown in the HA UI and logs
- ID format: `snake_case_short_description` (e.g. `lights_on_at_sunset`)

### Secrets
- Never hardcode tokens, passwords, coordinates, or URLs inline
- Add the value to `secrets.yaml` and reference it with `!secret key_name`
- `secrets.yaml` is gitignored

### File organisation
- One file per logical domain: `lighting.yaml`, `security.yaml`, `climate.yaml`, `presence.yaml`, etc.
- Keep each file focused — split when a file grows beyond ~10 automations

### Automation structure (required keys)
```yaml
- id: "unique_id"
  alias: "Human readable name"
  description: "What it does and why"
  trigger: [...]
  condition: []          # omit or leave empty if no conditions
  action: [...]
  mode: single           # single | restart | queued | parallel
```

### Script structure (required keys)
```yaml
script_name:
  alias: "Human readable name"
  sequence:
    - ...
```

## Home Assistant YAML quick reference

### Triggers
```yaml
- platform: state
  entity_id: binary_sensor.motion
  to: "on"

- platform: time
  at: "07:00:00"

- platform: sun
  event: sunset
  offset: "-00:30:00"

- platform: numeric_state
  entity_id: sensor.temperature
  above: 25

- platform: template
  value_template: "{{ states('sensor.foo') | float > 10 }}"
```

### Conditions
```yaml
- condition: state
  entity_id: input_boolean.guest_mode
  state: "off"

- condition: time
  after: "08:00:00"
  before: "22:00:00"
  weekday: [mon, tue, wed, thu, fri]

- condition: numeric_state
  entity_id: sensor.humidity
  below: 60
```

### Actions
```yaml
- action: light.turn_on
  target:
    area_id: living_room        # or entity_id / device_id
  data:
    brightness_pct: 80
    color_temp_kelvin: 3000

- action: switch.toggle
  target:
    entity_id: switch.garden_pump

- action: notify.mobile_app_my_phone
  data:
    title: "Alert"
    message: "Something happened"

- delay: "00:05:00"

- wait_template: "{{ is_state('binary_sensor.door', 'off') }}"
  timeout: "00:02:00"
```

### Templating (Jinja2)
```
{{ states('sensor.temperature') }}              # current state string
{{ states('sensor.temperature') | float }}      # cast to number
{{ is_state('binary_sensor.motion', 'on') }}    # boolean check
{{ state_attr('light.desk', 'brightness') }}    # attribute
{{ now().hour }}                                # current hour (0–23)
{{ now().strftime('%H:%M') }}                   # formatted time
{{ as_timestamp(now()) | int }}                 # unix timestamp
{{ trigger.to_state.state }}                    # state that fired trigger
```

## KNX notes

- KNX connection: auto-discovered (no gateway IP needed in config)
- All entities were UI-configured (unique_id prefix `knx_es_*`) — the YAML files here are the source-of-truth going forward
- ETS project has 117 group addresses; fetched via `knx/get_knx_project` WebSocket command
- Group address scheme:
  - `0/0/*` — Rollo kurz (step, DPT 1.007) → `move_short_address`
  - `0/1/*` — Rollo lang (move, DPT 1.008) → `move_long_address`
  - `0/2/*` — Rollo group commands (WZ alle, WZ Ausgang, etc.)
  - `0/3/*` — Rollo status (DPT 5.001) → `position_state_address`
  - `0/4/*` — Rollo Höhe / setpoint (DPT 5.001) → `position_address`
  - `1/3/*` — Temperature sensors (DPT 9.001)
  - `2/0/*`, `2/1/*`, `2/3/*` — Light switch commands (DPT 1.001)
  - `2/2/*` — Light status feedback (DPT 1.011)
  - `3/*`   — Gate (Tor) control and status
  - `4/*`   — Irrigation (Bewässerung) valves
- **TODOs in YAML**: `travelling_time_down/up` in covers.yaml need real measurements; climate setpoint addresses not in ETS export

## Deployment and Validation

### Unified deployment tool

Use `python3 -m deployment.skill` for full deployment (validation, git sync, automation push):

```bash
python3 -m deployment.skill deploy              # Full: validate → commit → push → sync automations
python3 -m deployment.skill validate            # Validate config only (no changes)
python3 -m deployment.skill push-automations    # Push all automations via REST API
python3 -m deployment.skill list-dashboards     # List all dashboards
python3 -m deployment.skill status              # Show HA connection status
```

**Options:**
```bash
python3 -m deployment.skill deploy --skip-git      # Skip git commit/push
python3 -m deployment.skill deploy --skip-reload   # Skip reloading automations
python3 -m deployment.skill deploy -v              # Verbose output
```

### Pushing individual dashboards

Use `push_dashboard.py` for individual dashboard operations:

```bash
# Push to HA (creates if missing, always overwrites config)
python3 push_dashboard.py dashboards/haussteuerung.yaml haus-steuerung "Haussteuerung"

# Validate before pushing (shows diff with live version)
python3 push_dashboard.py dashboards/bewaesserung.yaml garten-bewaesserung --check
```

**Arguments:**
- `<yaml_file>` — path to dashboard YAML
- `<url_path>` — URL path (must contain hyphen: `haus-steuerung`, `garten-bewaesserung`)
- `<title>` — dashboard title shown in UI
- `--check` — validate syntax and show diff with live version (no push)
- `--icon` — Mdi icon code (default: `mdi:home-variant`)

### Re-importing from HA

```bash
# Refresh entity catalog + automation stubs
python3 build_config.py

# Export / re-export dashboards
python3 export_dashboards.py
```

These scripts do not need `.env` export since they import from `deployment.Config`.

### Lovelace dashboard limitations

There is no server-side schema validation for Lovelace card configs. Card-level errors (e.g. missing required fields) only appear in the browser after pushing. The `--check` flag catches YAML syntax errors and shows structural diffs but cannot detect card schema violations.

To reload automations without full restart:
- HA UI → Developer Tools → YAML → Reload Automations
- Or: `python3 -m deployment.skill push-automations` to reload via API
- Or: `ha core restart` for a full restart

## Dashboard authoring rules

### Lovelace action syntax (HA 2024.8+)
Use `perform-action` — `call-service` is deprecated and causes config errors in HA 2026:
```yaml
tap_action:
  action: perform-action
  perform_action: cover.open_cover   # replaces deprecated "service:"
  target:
    entity_id: cover.some_cover
```

### Footer buttons in entities cards
The `buttons` footer row type **requires** an `entity` field on every button, even when the button uses only `tap_action`:
```yaml
footer:
  type: buttons
  entities:
    - entity: script.my_script       # required — omitting causes config error
      name: Run
      icon: mdi:play
      tap_action:
        action: perform-action
        perform_action: script.turn_on
        target:
          entity_id: script.my_script
```

## KNX covers — group addresses

Group covers in `knx/covers.yaml` (addresses in `0/2/*`) do **not** receive `position_state_address` feedback — those group state addresses never send KNX telegrams. Do not set `position_state_address` on group covers; without it HA uses assumed state (tracks last command) instead of showing "unknown".

## HA storage model — local YAML vs live entities

The HA instance does **not** read helpers or scripts from the local YAML files. These are **storage-based** (managed in HA's internal `.storage/`):

- `helpers/input_boolean.yaml`, `helpers/input_number.yaml`, `scripts/bewaesserung.yaml` are **documentation mirrors** only — HA does not load them
- Calling `input_boolean/reload` / `script/reload` services will **not** create new entities from these files
- To create new helpers or scripts, use the API (see section below)

You can confirm whether an entity is storage-based by checking `"editable": true` in its state attributes.

## Pushing new helpers and scripts via API

### input_boolean and input_number — WebSocket

Use `push_dashboard.py`'s WebSocket connection pattern:

```python
# input_boolean
{"type": "input_boolean/create", "name": "Bewässerung X aktiv", "icon": "mdi:water-outline"}
# ID is auto-derived from name: ä→a, spaces→_, lowercase

# input_number
{"type": "input_number/create", "name": "Bewässerung X Dauer", "icon": "mdi:timer-outline",
 "min": 1, "max": 60, "step": 1, "initial": 10, "unit_of_measurement": "min", "mode": "slider"}
```

### Scripts — REST API

```bash
export $(grep -v '^#' .env | sed 's/ *= */=/' | xargs)
# POST /api/config/script/config/{script_id}
curl -s -X POST "$HOME_ASSISTANT_URL/config/script/config/bewaesserung_X_manuell" \
  -H "Authorization: Bearer $HOME_ASSISTANT_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"alias":"...","icon":"mdi:watering-can-outline","mode":"single","sequence":[...]}'
# Returns {"result":"ok"} on success
```

Note: `script/config` WebSocket command is **read-only** (GET only, accepts `entity_id`).

### Irrigation script sequence pattern

Irrigation scripts send KNX telegrams directly — **not** `switch.turn_on/off`:

```yaml
sequence:
  - action: knx.send
    data:
      address: "4/1/7"        # valve group address (see knx/switches.yaml)
      payload: 1
      type: "1.001"
  - delay:
      minutes: "{{ states('input_number.bewasserung_X_dauer') | int }}"
  - action: knx.send
    data:
      address: "4/1/7"
      payload: 0
      type: "1.001"
```

## Helper/script naming conventions

Two coexisting naming schemes exist in this repo (historical artefact):
- **YAML-defined** (kreis-numbered): `bewaesserung_kreis_N_aktiv` / `bewaesserung_kreis_N_dauer` / `bewaesserung_kreis_N_manuell`
- **UI-created** (zone-name): `bewasserung_<zone>_aktiv` / `bewasserung_<zone>_dauer` / `bewaesserung_<zone>_manuell`

The Bewässerung dashboards use the **zone-name** format. New zones should follow the zone-name format. To add a new zone:
1. Create entities via API (see "Pushing new helpers and scripts via API" above)
2. Update the documentation mirrors: `helpers/input_boolean.yaml`, `helpers/input_number.yaml`, `scripts/bewaesserung.yaml`
3. Add the card to `dashboards/bewaesserung.yaml` and `dashboards/haussteuerung.yaml`
4. Push both dashboards

## Automation modes

| Mode | Behaviour |
|------|-----------|
| `single` | Ignore new triggers while running (default) |
| `restart` | Stop current run, restart from top |
| `queued` | Queue new triggers, run in order |
| `parallel` | Run multiple instances simultaneously |
